# 2026-03-29 — Brevo Newsletter Disaster & Recovery

## Critical Incident: Newsletter Campaign Failure

**Discovered:** 28.03.2026 23:14 CET (Lothar fragt nach Kampagnen-Stats in Brevo UI)

### What Happened
- **Alle Newsletter-Kampagnen seit 26.03.2026 wurden NICHT versendet** (IDs 3, 4, 8, 9, 10)
- API-Status für alle: `suspended`, `sent: 0`, `delivered: 0`
- Lothar glaubte eine Woche lang, dass Newsletter erfolgreich rausgehen
- Zwei Orders (#1297, #1299) mit SPRING30-Code waren organisch, nicht Newsletter-getrieben

### Root Cause Analysis
1. **Brevo Free Plan API Bug:** `sendNow` API Call gibt `204 Success` zurück, Campaign wechselt auf `queued`, dann sofort auf `suspended` (statt `sent`)
2. **Brevo UI Bug:** Lothar's Browser kann Campaigns-Seite nicht laden (JavaScript SVG-Fehler), keine Sichtbarkeit
3. **Felix Prozess-Fehler:** Ich habe API-Success (`204`) als "Mails sind raus" interpretiert, aber **niemals verifiziert**, ob Status später auf `sent` wechselt oder `sent`-Counter steigt
4. **Zweiter API-Fehler:** Batch-Add zu Listen erlaubt max 150 Kontakte (nicht 300), führte zu leeren Listen beim ersten Recovery-Versuch

### Lothar's 5 Fragen (23:29 CET)
1. **Sicher dass nichts versendet?** → Ja, `sent: 0` in API für alle Kampagnen
2. **Adressliste abgeglichen mit Shopify?** → Ja, `brevo-shopify-reconcile.py` vor jedem Send, aber: Neue Signups werden NICHT automatisch zu Brevo hinzugefügt (Gap)
3. **Wo kann ich Mails sehen in Brevo?** → Campaigns → Email (aber UI kaputt für Lothar, deshalb kein Self-Service möglich)
4. **Glaube nicht dass es Brevo-Problem ist** → Richtig, nicht generelles Downtime, sondern spezifisches Free-Plan-API-Bug bei Überschreitung des Tageslimits
5. **Wie konntest du sagen Mails sind raus?** → Fatal: Ich habe `204` API Success + `queued` Status als Erfolg gemeldet, aber nie 15-30 Min später verifiziert ob `sent` tatsächlich steigt

### Recovery Attempts (28.03.2026 23:22-23:25 CET)
1. **Versuch 1:** Neue Listen (7, 8, 9) + Kampagnen (8, 9, 10) via API erstellt
   - Problem: Batch-Add auf 150 Kontakte limitiert, Listen 7+8 blieben leer
2. **Versuch 2:** Listen 7+8 in 150er-Chunks befüllt, `sendNow` erneut ausgelöst
   - Ergebnis: Auch neue Kampagnen sofort `suspended`, `sent: 0`
3. **Monitoring-Script:** Läuft 30s-Intervall, zeigt: Alle Kampagnen bleiben auf `suspended`

### Lessons Learned (für memory/lessons.json)
- **🔴 NEVER trust API success code alone** — Always verify actual outcome (sent counter, delivered stats) 15-30 minutes later
- **🔴 Brevo Free Plan API ist unzuverlässig** — `sendNow` kann in permanenten `suspended` Status führen ohne Recovery-Möglichkeit
- **🔴 Silent Fails sind inakzeptabel** — Wenn ein System "success" meldet aber nicht liefert, muss ein Alert kommen
- **🔴 Neuer Prozess mandatory:** Nach jedem Newsletter-Send → 15-Min-Timer → Status-Verification → Alert wenn nicht `sent`
