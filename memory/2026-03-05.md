## ROOT CAUSE FOUND — session.dmScope
- Default `session.dmScope = "main"` collapses ALL DMs from ALL senders into one session
- `chat_id` in inbound metadata is only shown for the TRIGGERING message per turn
- QUEUED messages (arriving while Felix is processing) are appended WITHOUT their own chat_id
- This is why Chanté/Sina worked (their msg was the trigger = chat_id visible) but evening session failed (queued messages = no chat_id)
- Possible fix: change dmScope so each sender gets their own session → technical separation
- Shared full analysis with Lothar — he understands the technical issue now

## SOLUTION FOUND: session.dmScope = "per-channel-peer"
- OpenClaw has built-in "Secure DM Mode" — exactly what we need
- Options: "main" (default, all DMs shared), "per-peer", "per-channel-peer" (recommended), "per-account-channel-peer"
- "per-channel-peer" isolates each sender into their own session → no context leak, no confused responses
- Can also use `session.identityLinks` to merge same person across channels (e.g., Lothar on WhatsApp + Telegram)
- Awaiting Lothar's go-ahead to enable it — then allowlist can be safely re-added
